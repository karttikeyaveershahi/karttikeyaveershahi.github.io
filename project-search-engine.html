02 How It Works

When the program starts, it runs through a four-stage pipeline before the CLI becomes interactive:

docs.txt
→
Parser
→
Trie + Inverted Index
→
BM25 Scorer
→
Top-K Results
  1. Parser reads docs.txt, normalizes words, filters stop words, and loads documents into memory.
  2. Trie is built by inserting every word from every document, creating the inverted index.
  3. CLI starts and waits for user commands.
  4. On /search, query words are looked up in the Trie, BM25 scores are accumulated per document, sorted and returned.

03 Core Data Structures

Document

Represents one document loaded from docs.txt. Stores a numeric ID and a vector of normalized, filtered words.

// parser/document.h
struct Document {
    int id;                           // document ID (first token on each line)
    std::vector<std::string> words;   // normalized, filtered words
};

Posting

A single node in a postings list — the fundamental unit of the inverted index. Each word in the Trie has a linked list of Postings tracking which documents contain it and how often.

// index/posting.h
struct Posting {
    int docID;      // which document contains this word
    int termFreq;   // how many times the word appears in that document
    Posting* next;  // pointer to the next posting (linked list)
};

// Example postings lists:
"spears" → [docID:0, TF:5] → nullptr
"rapper" → [docID:1, TF:3] → [docID:7, TF:1] → nullptr

TrieNode

One node in the Trie. Every path from root to a leaf spells out a word. Leaf nodes hold the postings list for that word.

// index/trienode.h
struct TrieNode {
    TrieNode* children[26];  // one child per letter a–z
    Posting* postingsList;   // non-null only at end-of-word nodes
    bool isEndOfWord;
};

SearchResult

Holds one document's final BM25 score after a query is processed.

// search/search.h
struct SearchResult {
    int docID;
    double score;
};

04 Component Details

Parser

Reads docs.txt line by line, extracts the document ID from the first token, lowercases and strips non-alphabetic characters from each word, then filters stop words before storing. stopwords.h contains a hardcoded set of ~80 common English words excluded from the index entirely.

// stop word filtering during ingestion (parser.cpp)
auto stopWords = getStopWords();
// ...
if (!word.empty() && !isStopWord(word, stopWords))
    doc.words.push_back(word);

Trie + Inverted Index

The Trie is the inverted index — they are not two separate structures. Each leaf holds the postings list for that word.

  • Insertion — traverses character by character, creating nodes as needed. At the end node, increments termFreq if the doc already has a posting, otherwise prepends a new Posting node.
  • Search — traverses and returns the postings list at the end node, or nullptr if not found. Time complexity: O(M) where M = word length.
Why a Trie instead of a HashMap? A Trie supports prefix searches and autocomplete natively. It's also more memory-efficient when many words share common prefixes — those characters are stored once, not duplicated per key as in a HashMap.

05 BM25 Ranking

BM25 (Best Match 25) is the industry-standard ranking function used by search engines like Elasticsearch and Apache Solr.

// Score for a single word in a document
Score(doc, word) = IDF(word) × TF_normalized(doc, word)

// Inverse Document Frequency — penalizes common words
IDF(word) = log( (N − n + 0.5) / (n + 0.5) + 1 )

// TF with saturation + length normalization
TF_norm =           TF × (k₁ + 1)
            ──────────────────────────────────────────
            TF + k₁ × (1 − b + b × (docLen / avgLen))

// Multi-word query: accumulate scores across all terms
Score(doc, query) = Σ Score(doc, wordᵢ)
SymbolMeaning
NTotal number of documents
nNumber of documents containing the word
TFHow many times the word appears in this document
docLenTotal words in this document
avgLenAverage words per document across all docs
k₁ = 1.2TF saturation — controls diminishing returns on repetition
b = 0.75Document length normalization strength
  • IDF — penalizes common words. A word appearing in every document scores near 0; a rare word scores high.
  • TF normalization — rewards repetition with diminishing returns. 10 occurrences beats 1, but not by 10×.
  • Length normalization — prevents long documents from dominating simply by containing more words.

06 Search Engine Logic

The search() function runs in seven steps:

  1. Tokenize the query string into individual lowercase words
  2. Compute avgDocLen across all documents
  3. Build a docID → docLen map for fast lookup
  4. For each query word: look up in Trie → count n for IDF → walk postings list → compute BM25 → accumulate into scores map
  5. Convert scores map into a vector<SearchResult>
  6. Sort by score descending
  7. Trim to top-K and return

getPreview() returns the first 100 words of a document as a readable preview, appending "..." if truncated.

07 CLI Commands

startCommandLoop() runs an infinite loop parsing the first token of each line as a command:

10 Time Complexity

OperationComplexityNotes
ParsingO(N × L)N = docs, L = avg words/doc. One-time cost.
Index buildO(N × L × M)M = avg word length (~5 chars). One-time cost.
Trie lookupO(M)Per word. Extremely fast.
Search scoringO(Q × n)Q = query words, n = avg postings list length.
Result sortingO(D log D)D = number of matching documents.

11 Potential Improvements

PERF
Min-Heap for Top-K — replace full sort with a priority queue for O(N log K) instead of O(N log N) at scale.
STORAGE
Persistent Index — serialize the Trie to disk so the index doesn't need rebuilding on every run.
QUERY
Phrase Search — support "britney spears" as an exact phrase query using positional posting lists.
NLP
Stemming — treat "running", "runs", "ran" as the same token using a Porter Stemmer to improve recall.
SCALE
Skip Lists — speed up postings list traversal for very high-frequency words at large scale.
